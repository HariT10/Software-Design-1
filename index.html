Assignment 1

What is the software development process?

The software development process consists of many different stages. These many different stages are used to ensure that the final product is of the highest quality. The different stages are requirement gathering, planning and design, implementation and coding, the testing phase, and lastly, maintenance. The “requirement gathering” stage is when you ask lots of questions to get a clear understanding of how the client wants the program. The “planning/design” stage is where you would use all the gathered requirements from the previous step to create a project/game plan. This is also the step where you would plan/design the input, output, and algorithm to solve the problem. The “implementation/coding” stage is where you would program the software using the resources that you have gathered/designed. The “testing phase” is where the developer would do some test cases on the program to improve its quality. Finally, the “maintenance” stage is where you keep the program in perfect condition by doing updates, bug fixes, and making minor improvements that might be suggested by the user.

Why is software design important?

Software design is important because it creates a roadmap for solving the problem at hand and helps developers build quality software. It makes the whole task easier as it separates it into different parts. It also helps if there is a big team, each team member will understand their role. Software design also makes sure that the program will run well, be easy to maintain, and be easier to grow.

The requirement’s gathering stage is at the start of any software development process. This is the stage where the developers ask questions to figure out what the clients want in the software. As a team, you would write out the requirements that need to be in the software. Good requirements define the users, user terms, are categorized, prioritized, and flow logically. A good requirement example is “The software must run under Windows OS”. This is good because it specifically says the type of OS. A bad requirement example is “The software must run on our existing operating system”. This is bad because it doesn’t say the type of operating system. This stage is important for the entire software development process because it lays the basic foundations of what the entire project needs. Without a requirement gathering stage, it is more likely that you will face problems during the software development process.

Describe the Elements of Empathy

Empathy is the way of understanding and respecting others’ feelings, thoughts, and ways of life. The different elements of empathy are respect, understanding user experiences, a non-judgmental perspective, etc.

"Consumers, Literally" - Model 1

The three business models I picked are “Consumers, Literally”, “Sheep”, and “Human Factor”. Firstly, the problem with the “Consumers, Literally” model is that it views people as means of profit. A long-time internet industry analyst, Jerry Michalski, states that “businesses began to see a consumer as really no more than a gullet whose only purpose in life is to gulp products and crap cash”. Viewing customers this way is an explicit barrier to developing empathy for customers and users. This business model lacks respect, which is an element of empathy. This is because it does not view people as humans but rather as profit-making machines. Empathy requires respect and a connection between the consumer and seller, rather than just viewing customers as "profit makers".

"Sheep" - Model 2

Secondly, the problem with the “Sheep” model is that it is disrespectful to the people. Views customers as nothing but consumers and does not account for their feelings or experiences. This business model lacks the elements of empathy of respect and a non-judgmental perspective. This is because of the name “Sheep”, it sees everyone as similar when that’s not the case. Everyone is diverse and intelligent in different ways.

"Human Factor" - Model 3

Lastly, the problem with the “Human Factor” model is that it portrays people as knowing exactly what they want to achieve and following specific steps to get to their goals. The problem with this is that it does not show the bigger picture of human behavior. The element of empathy this model lacks is understanding the user's experiences and feelings. This is because this model focuses on goals and steps but not human emotions, feelings, and experiences.

Significance of Paper Prototyping

Prototyping in software design is creating a plan or mock-up design of the software application before it is developed. It is significant in the development process because it gives you a clear view of your software. The reasons why prototyping exists is to try out different designs, demonstrate concepts, and validate requirements. The more feedback designers get on their prototype, the more customers will like the final product. A quote from class by Donald A. Norman states that “designers know too much about their product to be objective judges”. This is important because, by showing more people, you can get better feedback from people other than the designers. Doing this will also make sure that there is more of a connection between the team.

3 Key Benefit's

The first key benefit of using prototyping in software development is feedback. Feedback is very important in improving the design of the final product. The second key benefit is that it encourages creativity. It encourages the team to get creative by making lots of different designs. The last key benefit is that it reduces development costs. This is because finding flaws in the prototyping stage saves more time and money than finding them in the development stage.

Basic Steps in Creating Paper Prototype

The first step in creating a paper prototype is to choose the most important user. This step is to identify the user or user persona. The second step is to determine some typical tasks. This step is to identify the tasks the users are supposed to perform when using the given software. The third step is to make screen shots or sketches. This step is to create different screen shots/sketches of the software; each screen has a different functionality. The fourth step is to perform usability tests. This is where you would have participants test your paper prototypes, navigate through different screen shots, and give feedback.
